# Как добавить алерт в Grafana через provisioning

## Зачем эта инструкция

Grafana alerting через provisioning — штука мощная, но документация размазана по десяткам страниц, а формат конфигов местами нелогичный. Здесь — минимум теории, максимум практики. Где формат продиктован историческими решениями разработчиков Grafana — принимаем как данность и идём дальше.

---

## Файл и его место

Все алерт-рулы лежат в одном файле:

```
monitoring/grafana/provisioning/alerting/rules.yml
```

Grafana читает его **только при запуске контейнера**. После изменений — пересоздаём контейнер:

```bash
docker compose up -d --force-recreate grafana
```

Обычный `restart` не всегда перечитывает provisioning.

---

## Структура файла

```yaml
apiVersion: 1

groups:
  - orgId: 1
    name: testplatform        # имя группы алертов
    folder: TestPlatform      # папка в Grafana UI
    interval: 1m              # как часто проверять
    rules:
      - uid: ...              # первый алерт
      - uid: ...              # второй алерт
```

Каждый `- uid:` — элемент списка. Все рулы должны начинаться с одинаковым отступом от левого края. Один лишний пробел — и YAML ломается.

## Анатомия одного алерта

Алерт состоит из 4 блоков. Разберём на примере мониторинга диска.

### Блок 1 — Заголовок

```yaml
- uid: alert-disk
  title: "Disk > 50%"
  condition: C
```

- `uid` — уникальный ID, придумываешь сам, не должен повторяться
- `title` — название, оно же придёт в Telegram
- `condition: C` — ссылка на блок данных с refId: C, который решает "алерт или нет"

### Блок 2 — Запрос к источнику данных (refId: A)

```yaml
data:
  - refId: A
    relativeTimeRange: {from: 300, to: 0}
    datasourceUid: PBFA97CFB590B2093
    model:
      expr: >-
        (1 - node_filesystem_avail_bytes{mountpoint="/"}
        / node_filesystem_size_bytes{mountpoint="/"}) * 100
      refId: A
```

- `refId: A` — имя этого блока данных. Указывается дважды: на верхнем уровне (для Alerting Engine Grafana) и внутри `model` (для плагина источника данных). Это дублирование — особенность архитектуры Grafana: два независимых слоя, каждый требует свой идентификатор. Принимаем как есть. [Документация →](https://grafana.com/docs/grafana/latest/alerting/set-up/provision-alerting-resources/file-provisioning/)
- `relativeTimeRange: {from: 300, to: 0}` — временное окно в секундах назад от текущего момента. `300` = 5 минут назад, `0` = сейчас. То есть "данные за последние 5 минут"
- `datasourceUid` — UID источника данных в Grafana. Для Prometheus и Loki — разные. Узнать свой: Grafana → Connections → Data sources → кликнуть на источник → UID в URL
- `expr` — запрос PromQL (для Prometheus) или LogQL (для Loki)

### Блок 3 — Условие срабатывания (refId: C)

```yaml
  - refId: C
    datasourceUid: __expr__
    model:
      type: classic_conditions
      refId: C
      conditions:
        - evaluator: {type: gt, params: [50]}
          operator: {type: and}
          query: {params: [A]}
          reducer: {type: last}
```

- `datasourceUid: __expr__` — встроенный вычислитель Grafana, не внешний источник
- `type: classic_conditions` — тип выражения. Самый простой и распространённый. Есть ещё `reduce`, `math`, но для большинства алертов хватает этого
- `evaluator: {type: gt, params: [50]}` — условие: `gt` = greater than (больше), `50` = порог. Другие варианты: `lt` (меньше), `within_range`, `outside_range`. Формат `{type: ..., params: [...]}` — фиксированная структура Grafana, писать иначе нельзя. [Документация →](https://grafana.com/docs/grafana/latest/alerting/set-up/provision-alerting-resources/file-provisioning/#export-alert-rules)
- `operator: {type: and}` — логический оператор для нескольких условий. С одним условием ни на что не влияет, но Grafana требует его указать
- `query: {params: [A]}` — откуда брать данные: из блока с refId: A
- `reducer: {type: last}` — какое значение взять: последнее. Варианты: `avg`, `min`, `max`, `sum`

### Блок 4 — Поведение и аннотации

```yaml
  for: 2m
  noDataState: OK
  execErrState: Error
  annotations:
    summary: "Диск заполнен > 50%"
  isPaused: false
```

- `for: 2m` — условие должно держаться 2 минуты подряд, прежде чем алерт сработает. Защита от ложных срабатываний. Пока ждёт — состояние `Pending`
- `noDataState` — что делать, если данных нет. `OK` = не паниковать, `Alerting` = считать как срабатывание
- `execErrState` — что делать, если запрос к Prometheus упал. `Error` = состояние ошибки, `OK` = игнорировать, `Alerting` = считать как алерт
- `annotations.summary` — текст, который придёт в Telegram. Поддерживает шаблоны: `{{ $labels.job }}`
- `isPaused` — `false` = алерт активен, `true` = алерт на паузе (не проверяется, не шлёт уведомления, но остаётся в списке)

## Пошаговое добавление нового алерта

**1. Открыть файл:**

```bash
nano monitoring/grafana/provisioning/alerting/rules.yml
```

**2. Дописать новый рул в конец списка `rules:`.** Отступ дефиса `-` должен совпадать с остальными рулами.

**3. Проверить YAML-синтаксис:**

```bash
python3 -c "import yaml; yaml.safe_load(open('monitoring/grafana/provisioning/alerting/rules.yml')); print('OK')"
```

**4. Пересоздать Grafana:**

```bash
docker compose up -d --force-recreate grafana
```

**5. Проверить логи:**

```bash
docker compose logs grafana --tail=30
```

**6. Убедиться в Grafana UI:** Alerting → Alert rules → найти новый алерт.

## Шаблон для копирования

```yaml
      - uid: alert-НАЗВАНИЕ
        title: "Описание алерта"
        condition: C
        data:
          - refId: A
            relativeTimeRange: {from: 300, to: 0}
            datasourceUid: PBFA97CFB590B2093
            model:
              expr: PROMQL_ЗАПРОС
              refId: A
          - refId: C
            datasourceUid: __expr__
            model:
              type: classic_conditions
              refId: C
              conditions:
                - evaluator: {type: gt, params: [ПОРОГ]}
                  operator: {type: and}
                  query: {params: [A]}
                  reducer: {type: last}
        for: 2m
        noDataState: OK
        execErrState: Error
        annotations:
          summary: "Текст для Telegram"
        isPaused: false
```

Заменить `НАЗВАНИЕ`, `PROMQL_ЗАПРОС`, `ПОРОГ` и `summary` — готово.

## Частые ошибки

- **Кривые отступы** — YAML ломается от одного лишнего пробела. Всегда проверять через `python3 -c "import yaml; ..."`
- **restart вместо force-recreate** — `docker compose restart grafana` не всегда перечитывает provisioning-файлы
- **Команды из корня** — `docker compose` работает только из папки с `docker-compose.yml`. Всегда `cd ~/testplatform` перед работой
- **Дублирование refId** — верхний и нижний `refId` должны совпадать. Не удалять ни один из них
